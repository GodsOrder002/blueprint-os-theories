有的，而且你描述的这条路（“轻量封装 + 多账户 + 审计 + 类 SQL 的操作感”）基本可以拆成两层：**上层给人用（UI/权限/文件）**，**下层做硬审计（SQL/连接/变更日志）**。

## 1) 更像“轻量定制系统”的开源：把 MySQL 二次封装成可管可控的后台

### Directus（很贴你说的“比分布式平台轻”）

* **它的定位**：把现有数据库（含 MySQL）变成“带账号体系、权限、可操作 UI + API”的数据管理后台。
* **审计**：自带 activity feed / accountability，用来追踪数据变更与操作者。([Directus][1])
* **文件**：有文件/文件夹能力；文件信息作为常规集合存储，因此**文件权限可按用户/角色细粒度配置**。([Directus][2])

> 这基本就是你要的“轻量定制化数据+文件管理 + 多账号 + 可追溯”。

### NocoDB（像 Airtable 的“表格化封装”）

* **优势**：对非 SQL 人群更友好；自带角色/权限体系。([NocoDB][3])
* **但要注意**：他们文档明确写了 **Workspace audit logs 仅企业版提供**（包括自托管企业版/云企业版）。([NocoDB][4])

> 所以它能做“封装+多账户”，但“审计”这件事开源自托管是否够用，要看你接受不接受企业版边界。

## 2) 更像“轻量审计底座”的开源：不改你的工具习惯，但把 SQL 行为纳入审计

### ProxySQL（把它当数据库“统一入口/门卫”）

* 有 **Audit Log** 能记录连接活动（谁来连、怎么连）。([ProxySQL][5])
* 还有 **Query Logging** 能记录经过代理的 SQL（可全量或筛选）。([ProxySQL][6])

> 这个思路很“轻”：你照样用 MySQL/SQL，但所有访问必须走代理，于是审计更像“不可绕过”。

### Percona Server for MySQL 的 Audit Log Plugin（偏“数据库侧审计插件”）

* 明确提供连接与查询活动的监控与日志，是 MySQL 企业审计插件的开源替代路径之一。([Percona Documentation][7])

## 3) 你说的“两件事”里，第一个“类 SQL 语言”怎么落在“用我处理文件”上

你这个想法非常对：你需要一个**稳定的指令表达方式**，来描述：

* 这个文件怎么处理、那个文件怎么处理
* 多文件怎么“联动”（等价于 join key / 匹配规则）
* 输出要什么结构（等价于 select 列表）

不用真的发明新语言，很多人会用“固定格式的自然语言 DSL”来做到“像写 SQL 一样可复用、可审计”。

## 4) 如果你要我给一个“最轻、最接近你描述”的答案

* **想要“数据 + 文件 + 权限 + 审计”一体化、又不重**：优先看 **Directus**。([Directus][1])
* **想保留纯 SQL 工作流，但把审计做硬**：看 **ProxySQL**（入口审计）+（可选）Percona 审计插件（库侧审计）。([ProxySQL][5])
